# Word 排版规范

本文档定义行程手册 Word 文档的视觉标准和 JSON 数据结构。

## 文档规格

| 属性 | 值 |
|------|------|
| 纸张 | A4 |
| 页边距 | 上下各 2cm，左右各 2.5cm |
| 正文字体 | 微软雅黑 小四（12pt） |
| 小标题字体 | 微软雅黑 四号（14pt） |
| 标题颜色 | #1B5E7B 深蓝 |
| 强调色 | #2E86AB 海洋蓝 |
| 费用色 | #C73E1D 红色 |

**字号说明**：正文内容统一使用小四（12pt），包括活动描述、住宿信息、贴士、表格文字等；所有标题（H1/H2/H3）统一使用四号（14pt），包括章节标题和每日行程标题。封面标题、预算总计等特殊强调元素使用更大字号，不受此限制。

## 文档结构

1. **封面** — 标题 + 副标题 + 日期/人数/目的地（居中排列）
2. **行程概览** — Day 1~N 的日期+主题列表 + 预估总费用
3. **每日行程** — Day 标题 + 住宿 + 上午/下午/晚上分段活动
4. **预算明细** — 分类费用表格 + 总计 + 人均
5. **实用贴士** — 行李清单 + 注意事项 + 紧急联系
6. **尾页** — 祝福语 + 免责声明

## 活动分类

| 分类 | 颜色 | 用途 |
|------|------|------|
| transport | #2E86AB 蓝 | 交通 |
| sight | #F18F01 橙 | 景点 |
| food | #C73E1D 红 | 餐饮 |
| hotel | #27AE60 绿 | 住宿 |
| shopping | #8E44AD 紫 | 购物 |
| activity | #E67E22 深橙 | 活动 |

## 数据质量要求

1. **time 字段**：可选，格式 `HH:MM`（如 `08:00`、`14:30`）。**仅用于排序和上午/下午/晚上分段，文档中不显示具体时刻。**
2. **cost 字段**：字符串，**必须含 ¥ 符号或为"免费"/"¥0"**，如 `"¥55"`、`"¥120/人"`、`"打车¥35"`、`"免费"`、`"¥0（65岁免票）"`。禁止写纯数字 `"55"`。优惠金额根据 `ticket-discounts.md` 中的政策和用户优惠身份计算。
3. **duration 字段**：字符串，用中文格式，如 `"2小时"`、`"1.5-2小时"`、`"30分钟"`。
4. **rating 字段**：字符串，用星级符号格式，如 `"★★★★★"`、`"★★★★☆"`。
5. **detail 字段**：建议控制在 100 字以内。
6. **category**：必须从 6 个枚举值中选（transport/sight/food/hotel/shopping/activity），否则显示默认颜色。
7. **accommodation 去重**：如果连续多天住同一酒店，每天的 accommodation 仍填写完整信息，但 `budget.accommodation.items` 只写一次并备注晚数，如 `"XX酒店 ¥250/晚 × 2晚"`。
8. **accommodation.price**：填写搜索到的**实际价格**（用户在第 6 步已确认），不是固定价格。SKILL.md 推荐阶段"不写死价格"指的是不要凭记忆假设价格，但确认行程后 JSON 中需要填入搜索到的真实价格。
9. **accommodation.room_plan**：填写第 1 步用户确认的房间方案（如"1间大床房"、"2间（1大床+1双床）"）。每天住宿都要填写（同一酒店则重复）。Word 文档在住宿信息行展示。
10. **交通类 activity 的 detail**：返程当天的交通活动应在 detail 中包含时间预留提醒，如 `"提前 1h 到站，高铁约 4.5h，建议下午 2 点前出发去车站"`。
11. **budget.items**：每项控制在 40 字以内，超过 4 项截断显示。
12. **budget.grand_total**：必须等于各分类 total 之和。
13. **budget.other**：必须包含一条"弹性预留 ¥XXX"（总预算的 5-15%），覆盖临时打车、小吃、纪念品等不可预见开销。其余明细为搜索到的实际费用（如特产、体验费用）。
14. **packing**：建议 9-15 项。
15. **tips.notes**：必须包含行程草案中的策略判断信息（如"Day 2 下午有雷阵雨，户外景点已排上午"、"高海拔地区第一天避免剧烈运动"），这些信息在 JSON 中没有结构化字段，只能存放在 notes 中。普通注意事项（如"带防晒"）和策略判断都要有。
16. **disclaimer**：应覆盖以下维度，控制在 2-3 句话内：
    - **价格时效性**：门票、交通等价格可能随季节调整，出发前以官方为准（必写）
    - **AI 非专业服务**：本行程由 AI 辅助生成，不构成专业旅行建议（必写）
    - **实时变动因素**：天气、客流、景区维护等可能影响实际体验（必写）
    - **高风险活动追加**：行程含漂流/登山/潜水等 → 追加"XX 类活动有安全风险，请根据自身状况量力而行"
    - **特殊目的地追加**：高海拔 → 追加"高海拔地区注意高原反应，如有不适及时就医"；出境 → 追加"请关注目的地签证和安全形势"
17. **weather**：可选字段，用于在文档行程概览页展示天气摘要。
    - `high` 和 `low` 是**数字**（不写 °C 或 "度"），如 `"high": 35` 而非 `"35°C"`
    - `rain` 无降雨时填 `"无"`，有降雨时描述（如 `"偶有雷阵雨"`、`"持续阴雨"`）
    - `dressing` 给具体建议（如 `"短袖 + 防晒 + 折叠伞"`，而非笼统的"夏季着装"）
    - `risk` 写天气对行程的具体影响（如 `"下午可能有雷阵雨，户外活动建议排上午"`），与 `tips.notes` 不冲突——前者给用户看，后者记录决策依据
18. **transit 字段**：可选，描述从上一个活动到当前活动的交通方式。
    - 每天的**第一个活动**可省略 transit（除非从酒店出发需要标注交通）
    - 后续每个活动（景点、餐饮、购物等）**必须有 transit**，`options` 至少 2 项
    - 每个 option 包含：`mode`（交通方式，带 emoji 如 🚗🚌🚲🚶🚕）、`duration`（耗时）、`cost`（费用，可选）
    - `recommended: true` 标记推荐方案，每个 transit 只能有 1 个推荐
    - `recommendation` 可选，简短说明推荐理由（≤30字）
    - `mode` 中的 emoji 统一使用：🚶 步行、🚲 骑行、🚌 公交/班车、🚇 地铁、🚕 打车、🚗 包车/自驾
19. **extra_restaurants 字段**：可选，5-10 家备选餐厅，显示在文档底部作为灵活选择。
    - 行程中的餐厅是交通最方便的（紧邻当日景点），这里的备选覆盖不同菜系和区域
    - 每家必须包含：`name`、`cuisine`、`price`（人均，含 ¥ 符号）、`address`、`recommendation`（≤30字）
    - 推荐数量：最少 5 家，最多 10 家
    - 不要与行程中已有的餐厅重复
20. **route 字段**：可选，仅用于**大型景区**（`duration ≥ 2小时` 且 `category = sight`）。
    - `must_see`：数组，3-6 个景区内必看子景点名称，搜索 `"{景区名} 必看 攻略 路线"` 获取
    - `path`：推荐游览路线，用 `→` 连接各节点，控制在 100 字以内（如 `"入口 → 西山石窟（1.5h）→ 漫水桥 → 东山石窟（40min）→ 香山寺（30min）→ 白园（20min）→ 出口"`）
    - `skip`：可跳过部分，50 字以内（如 `"东山石窟较小，时间紧可跳过，直接去香山寺"`）
    - 小型景点（duration < 2h）不需要 route，只写 detail 即可
    - Word 文档中 route 用浅色背景框 + 🗺 标题渲染，与 detail 视觉上区分
